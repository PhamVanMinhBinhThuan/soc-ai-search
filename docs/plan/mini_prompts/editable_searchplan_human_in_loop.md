# Mini Prompt - Editable SearchPlan Human-in-the-loop

Hãy triển khai tính năng cho phép SOC analyst review và chỉnh sửa `SearchPlan` sau khi LLM sinh ra, nhưng vẫn giữ `generated_dsl` read-only và chỉ do backend compiler sinh ra.

Mục tiêu của tính năng này là trả lời rõ câu hỏi khi bảo vệ:

> Nếu LLM sinh sai SearchPlan thì sao?

Câu trả lời mong muốn:

> LLM không được query Elasticsearch trực tiếp. LLM chỉ sinh SearchPlan trung gian. Analyst có thể review/chỉnh SearchPlan. Backend luôn validate lại SearchPlan bằng allowlist/guardrail rồi mới compile sang Elasticsearch DSL. DSL là read-only, user/LLM không thể sửa hoặc gửi DSL tùy ý.

## Nguyên tắc thiết kế

1. Cho phép edit **SearchPlan** vì SearchPlan là contract nghiệp vụ gần natural language.
2. Không cho phép edit **Elasticsearch DSL**.
3. Không cho frontend tự compile DSL.
4. Không cho user gửi DSL tùy ý.
5. Khi user chỉnh SearchPlan và bấm Run, frontend phải gọi backend endpoint kỹ thuật `POST /api/v1/search/plan`.
6. Backend phải validate lại SearchPlan như mọi request khác.
7. Compiler backend sinh DSL mới.
8. UI cập nhật lại kết quả, SearchPlan final và generated DSL mới từ response backend.
9. Nếu SearchPlan invalid, không query Elasticsearch và hiển thị lỗi rõ trên UI.

Luồng mong muốn:

```text
Natural language search
        ↓
LLM sinh SearchPlan
        ↓
Backend validate + execute lần đầu
        ↓
UI hiển thị Validated SearchPlan + Compiled DSL
        ↓
Analyst bấm Edit SearchPlan
        ↓
Analyst sửa JSON SearchPlan
        ↓
Analyst bấm Validate / Run Edited Plan
        ↓
Frontend gửi SearchPlan tới POST /api/v1/search/plan
        ↓
Backend validate lại
        ↓
Compiler sinh DSL mới
        ↓
Elasticsearch execute
        ↓
UI cập nhật result + generated_dsl mới
```

## Scope bắt buộc

### 1. Frontend: Editable SearchPlan panel

Trong `QueryTransparency` hoặc component tương đương:

- Tab `Validated SearchPlan` hiện tại đang render JSON read-only.
- Thêm chế độ edit cho SearchPlan:
  - Nút `Edit SearchPlan`.
  - Khi edit, hiển thị JSON editor/textarea dùng dark theme hiện có.
  - Editor phải có realtime JSON syntax validation khi user đang gõ để tránh gọi API với JSON sai cú pháp.
  - MVP có thể dùng `textarea` + `JSON.parse` theo `onChange` hoặc debounce nhẹ. Chỉ dùng CodeMirror/Monaco nếu project đã có sẵn hoặc thêm được gọn, không làm phình build/test. Không bắt buộc cài Monaco.
  - Có nút `Validate` nếu triển khai validate local được an toàn, hoặc có thể bỏ qua validate local và chỉ validate bằng backend khi Run.
  - Có nút `Run Edited Plan`.
  - Có nút `Reset to AI Plan` hoặc `Cancel` để quay lại SearchPlan đang active.
  - Nút `Reset to AI Plan` phải khôi phục chính xác JSON gốc do LLM/backend trả về cho câu query hiện tại, xóa toàn bộ chỉnh sửa draft của user. Không reset về bản edited đã chạy thành công trước đó trừ khi đó chính là SearchPlan gốc của query mới.
- Editor phải format JSON đẹp bằng `JSON.stringify(searchPlan, null, 2)` khi mở edit.
- Nếu JSON parse lỗi ở frontend, hiển thị lỗi ngay:
  - `SearchPlan JSON is invalid`
  - Không gọi backend.
- Nếu backend trả 400, hiển thị lỗi validation từ backend:
  - ví dụ field ngoài allowlist, mode không hợp lệ, top_n > 100, v.v.
- Khi backend trả success, cập nhật response hiện tại trong `App.tsx` giống như search response bình thường.

### 2. DSL read-only

Tab `Compiled DSL` / `generated_dsl`:

- Vẫn read-only.
- Không thêm editor cho DSL.
- Có badge hoặc text rõ:

```text
Read-only · Generated by backend compiler
```

- Giữ nút Copy nếu đang có.
- Không cho user paste/submit DSL.

### 3. Gọi lại backend bằng `/api/v1/search/plan`

Tạo hoặc mở rộng frontend service, ví dụ:

```ts
runSearchPlan(searchPlan: SearchPlanDto, signal?: AbortSignal): Promise<SearchPlanResponseDto>
```

Yêu cầu:

- Gọi `POST /api/v1/search/plan`.
- Body chính là SearchPlan đã chỉnh.
- Có Authorization header như các API khác.
- Không gọi `/api/v1/search` cho edited SearchPlan vì endpoint này dành cho natural language.
- Không gọi Elasticsearch trực tiếp từ frontend.

Nếu DTO response của `/api/v1/search/plan` khác response của `/api/v1/search`, hãy normalize ở frontend để UI dùng chung được:

- `mode`
- `search_plan`
- `generated_dsl`
- `total`
- `page`
- `size`
- `total_pages`
- `events`
- `aggregation_results`
- `chart_metadata`
- `aggregation_type`

Nếu backend response chưa đủ field để UI reuse, hãy sửa backend response nhẹ nhàng nhưng không phá contract cũ.

### 4. Pagination guardrail khi run edited SearchPlan

Nếu user chỉnh `page` hoặc `size` trong SearchPlan:

- Cho phép backend validate theo rule hiện có.
- `size` vẫn tối đa 100.
- UI nên giữ default/result page size hiện tại là 10 nếu SearchPlan không có size.
- Không để frontend tự nâng size quá 100.

Nếu implementation hiện tại đang dùng `SearchPlan.size`, hãy giữ đúng contract backend.

### 5. RBAC

Áp dụng permission rõ:

- `SOC_VIEWER`: chỉ được xem SearchPlan/DSL, không thấy hoặc bị disable nút edit/run.
- `SOC_ANALYST`: được edit SearchPlan và run edited plan.
- `SOC_ADMIN`: được quyền như analyst nhờ role hierarchy/permission helper.

Dùng permission helper hiện có trong `frontend/src/auth/permissions.ts`. Không hardcode role check rải rác trong component nếu đã có helper pattern.

UI message cho viewer:

```text
SearchPlan editing requires SOC_ANALYST or SOC_ADMIN.
```

### 6. Audit/history behavior

Nếu backend hiện tại audit/search history chỉ được tạo khi gọi `/api/v1/search`, hãy kiểm tra `/api/v1/search/plan` hiện có có audit hay không.

Không bắt buộc phải thêm audit mới nếu task quá lớn, nhưng nếu hệ thống hiện đã audit `/api/v1/search/plan`, thì edited SearchPlan run cũng phải lưu đúng query history/audit.

Nếu có thể làm nhỏ gọn, hãy lưu thêm dấu hiệu trong response/history/audit rằng đây là edited technical SearchPlan, ví dụ:

- `original_question` có thể là `Edited SearchPlan` nếu endpoint kỹ thuật không có question.
- Không phá schema hiện tại.

Không triển khai audit refactor lớn trong task này.

### 7. Diff giữa AI SearchPlan và edited SearchPlan

Nếu làm được gọn, thêm phần hiển thị thay đổi đơn giản:

```text
Edited fields:
- filters.severity: null -> ["critical"]
- filters.user: null -> "admin"
```

Nếu phức tạp, có thể bỏ qua. Ưu tiên edit/run/validation trước.

### 8. UX states

Nếu user đang ở edit mode và có thay đổi chưa chạy:

- Không làm mất draft khi user chỉ chuyển qua lại giữa tab `Validated SearchPlan` và `Compiled DSL`.
- Nếu user gõ query mới, chọn suggested query mới, chạy history item mới hoặc rời khỏi màn hình khiến draft bị mất, hãy hiển thị confirm/cảnh báo nhỏ hoặc giữ lại draft state an toàn.
- Không cần modal phức tạp; dùng browser confirm hoặc inline warning là đủ cho MVP.

Khi user bấm `Run Edited Plan`:

- Hiển thị loading state trên button.
- Disable button trong lúc request chạy.
- Nếu success:
  - thoát edit mode;
  - cập nhật result table/chart;
  - cập nhật SearchPlan view;
  - cập nhật Compiled DSL view;
  - hiển thị một message nhỏ: `Edited SearchPlan executed successfully`.
- Nếu fail:
  - giữ editor mở;
  - hiển thị lỗi rõ;
  - không xóa nội dung user đang sửa.

### 9. Backend safety check

Không được thêm endpoint nhận DSL tùy ý.

Nếu sửa backend, chỉ sửa để hỗ trợ tốt hơn `/api/v1/search/plan` hoặc response shape.

Backend vẫn phải reject:

- field ngoài allowlist;
- `.keyword` do user nhập;
- `script`, `wildcard`, `query_string`;
- aggregation invalid;
- `size > 100`;
- mode không hỗ trợ;
- message_query trong aggregation nếu rule hiện tại reject.

Không làm yếu validator để UI edit dễ hơn.

## Test bắt buộc

Thêm frontend tests phù hợp:

1. `SOC_VIEWER` không thấy hoặc không dùng được nút `Edit SearchPlan`.
2. `SOC_ANALYST` thấy nút `Edit SearchPlan`.
3. Click `Edit SearchPlan` hiển thị JSON editor với SearchPlan hiện tại.
4. JSON invalid hiển thị lỗi và không gọi API.
5. Click `Run Edited Plan` gọi `POST /api/v1/search/plan` qua service mới.
6. Backend mock success cập nhật result và generated DSL mới.
7. Backend mock 400 giữ editor mở và hiển thị lỗi.
8. DSL tab vẫn read-only và không có nút edit DSL.
9. Nếu aggregation SearchPlan edited success, UI cập nhật chart/table đúng.
10. Nếu search SearchPlan edited success, UI cập nhật raw events đúng.
11. Security bypass test: cố tình chèn field nguy hiểm như `script`, `query`, `aggs`, `query_string`, hoặc field filter lạ như `password` vào edited SearchPlan rồi run qua `/api/v1/search/plan`. Assert backend trả `400 Bad Request` hoặc lỗi validation có kiểm soát, và UI không cập nhật `generated_dsl` mới từ payload độc hại.
12. Draft SearchPlan không bị mất khi user chuyển giữa tab SearchPlan và DSL.
13. `Reset to AI Plan` khôi phục đúng SearchPlan gốc của query hiện tại.

Thêm service tests nếu project đang có pattern:

- `runSearchPlan` gửi đúng URL `/api/v1/search/plan`.
- Có Authorization header thông qua api-client hiện có.
- Error response map thành UI error đúng.

Backend tests chỉ thêm nếu phải sửa backend:

- Edited SearchPlan endpoint vẫn validate allowlist.
- Invalid SearchPlan trả 400.
- Response generated_dsl là object/map, không phải string.
- DSL vẫn do compiler sinh ra.

## Demo script sau khi làm

Sau khi triển khai, mô tả cách demo:

1. Login analyst.
2. Search câu:

```text
Show me failed login attempts from China in the last 24h
```

3. Mở `Validated SearchPlan`.
4. Bấm `Edit SearchPlan`.
5. Đổi field, ví dụ:

```json
"severity": ["critical"]
```

hoặc đổi:

```json
"country_code": ["VN"]
```

6. Bấm `Run Edited Plan`.
7. UI cập nhật kết quả và DSL mới.
8. Thử nhập field xấu:

```json
"password": "abc"
```

9. Backend trả lỗi validation, không query Elasticsearch.
10. Chuyển sang tab `Compiled DSL`, nhấn mạnh DSL chỉ read-only.
